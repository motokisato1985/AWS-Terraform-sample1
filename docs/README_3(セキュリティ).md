# セキュリティ設計方針


## 1. WAF (Web Application Firewall)
- CloudFrontへの適用：CloudFront に WAF を適用し、日本国外からのアクセスをブロックする。
- 検証環境(dev)：社内IPまたは許可したIPアドレスのみアクセス可能とする。
- 本番環境(prod)：日本国内からのアクセスを前提とし、AWSマネージドルールおよびレート制限により保護する。
- 脆弱性対策：AWSマネージドルールを導入し、SQLi/XSSなどの一般的な攻撃への対策を行う。
- レート制限：短時間の大量リクエストに対して、IP単位で遮断する設定を行う  

※AWSマネージドルールの更新状況は定期的に確認し、必要に応じてルールの有効化・調整を行う。  

---

## 2. ネットワークセキュリティ
- 多層防御：セキュリティグループを連鎖させ、後続のサービス(ECS, RDS)は前段のSGからの通信のみを許可する最小権限設計。
- プライベート配置：RDSおよびECSタスクはプライベートサブネットに配置し、パブリックIPを持たせないことで直接攻撃のリスクを排除。
---

## 3. 異常検知・DDoS防御
- 自動防御：AWS Shield Standardによる基盤層の防御を有効化。
- 監視：CloudWatch Alarmsにより、ALB 5xxやECS・RDSのメトリクス異常を検知し、サービス影響の早期把握を行う。
### フロー
 1. WAFにより異常トラフィックを自動遮断
 2. CloudWatch Alarmsによりサービス影響の兆候を検知
 3. SNS通知により運用者へ即時通知
 4. CloudWatch Dashboardで影響範囲を確認
 5. 必要に応じてWAFルールの調整やIPブロックを追加
---

## 4. 監視・通知
- 可用性監視：CloudWatch Alarmsにより、ALB 5xx、ECSのCPU/Memory使用率、稼働タスク数、RDSのCPU/空き容量を監視する。
- アプリ異常検知：CloudWatch Logs のメトリクスフィルタにより、アプリケーションログ内のERROR / Exception を検知して通知する。
- 通知：異常発生時は SNS を通じて通知し、初動対応を迅速化する。
- 可視化：CloudWatch Dashboardに主要メトリクスを集約し、障害時の切り分けを容易にする。 
---

## 5. セキュリティホール発見時の対応フロー

 1. 検知
 - CloudWatch Alarms / CloudWatch Logs / アプリログから異常を検知
 - または脆弱性診断・外部報告により発見

 2. 影響範囲の特定
 - CloudWatch Dashboardおよびログを確認し、影響範囲（ECS / RDS / 特定エンドポイント）を特定

 3. 一時対策（緊急対応）
 - WAFルールにより対象リクエストの遮断
 - 必要に応じて特定IPのブロック
 - ECSタスクの再起動やスケールアウトでサービス継続性を確保

 4. 恒久対応
 - アプリケーションコードの修正（SQLインジェクション対策など）
 - セキュリティ設定の見直し（SG / IAM / WAFルール）
 - 修正後、CI/CDにより再デプロイ

 5. 再発防止
 - CloudWatch Alarmsの閾値や監視項目の見直し
 - ログ監視ルールの強化
 - 必要に応じてWAFマネージドルールの追加
---

## 6. シークレット管理
- Secrets Managerの利用：本番環境では、APP_KEY や DBパスワードなどの秘匿情報を AWS Secrets Manager で管理する。
- 平文管理の回避：Terraformコード、タスク定義、環境変数へ機密情報を直接記載しないことで、漏えいリスクを低減する。
- RDSとの関係：ECSタスクは Secrets Manager 経由で RDS 接続用パスワードを取得し、その認証情報を用いて RDS(MySQL) に接続する。
- 運用上の意図：アプリケーションとデータベースの間で必要となる認証情報を安全に受け渡すための仕組みとして採用している。
---

## 7. セキュリティグループの設計について

本環境では、不要な通信を防ぐためインバウンド通信を最小限に制限している。
- ALB：CloudFrontからのHTTPSのみ許可
- ECS：ALBからのHTTP(内部通信)のみ許可
- RDS：ECSからのMySQL接続のみ許可
---

## 8. アウトバウンド通信について

ECSおよびCodeBuildのアウトバウンド通信は、現時点では `0.0.0.0/0` に対して全許可としている。

理由は以下の通り：

- ECS：
  - ECRからのコンテナイメージ取得
  - CloudWatch Logsへのログ送信
  - SSM Parameter Store / Secrets Managerからの設定取得
- CodeBuild：
  - ECRへのイメージPush/Pull
  - S3（Artifact Store）へのアクセス
  - CloudWatch Logsへのログ出力
  - GitHubとの通信

これらの通信先はAWSサービスに広く分散しており、検証環境では柔軟性を優先して全許可としている。
