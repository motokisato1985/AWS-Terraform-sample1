# ECS / RDS 性能選定・スケーリング戦略

本ドキュメントでは、システム構成におけるリソースの選定根拠と、負荷状況に応じた改善方針を定義します。


## 1. 前提
- 軽量なLaravelアプリを前提とし、主なアクセスは「店舗一覧表示」「店舗詳細検索」「レビュー閲覧」などを想定
- 日本国内限定で、急激な同時アクセスは少ない
- 画像やCSS/JSはCloudFrontで配信されるため、ECS/RDSへの負荷は軽減される
- 動的処理は一覧取得・詳細取得・検索・ログイン程度
- 1リクエストあたりの処理時間を約300〜500msと仮定すると、
　1タスクあたり約2〜5 requests/secの処理が可能と見積もった。


## 2. ECSの性能選定

### 検証環境
- タスクサイズ: 0.25vCPU / 1GB
- タスク数: 通常1、最大2（AutoScaling）
- 想定処理能力:1タスクあたり 1〜3 requests/sec
- 最大 2〜6 requests/sec 程度
- 軽負荷なLaravelアプリを前提とし、機能確認・デプロイ確認用途として十分な性能を持つ構成とした。
　(簡易的な一覧取得・詳細表示中心のため、CPU/メモリ負荷は低いと想定)
- CPU使用率70%を閾値としてスケールアウトする設定とし、必要に応じてタスク数を増加させる構成とした。
　(動作確認用)

### 本番環境
- タスクサイズ: 0.5vCPU / 2GB
- タスク数: 通常2、最大3（AutoScaling）
- 想定処理能力:1タスクあたり 2〜5 requests/sec
- 最大 6〜15 requests/sec 程度
- 冗長性確保のため常時2タスク稼働とし、負荷増加時には最大3タスクまでスケールアウト可能とした。
- CPU使用率70%を閾値としてスケールアウトを行う。
- ECSは横方向スケール（タスク増加）により負荷増加に対応可能とした。


## 3. RDSの性能選定

### 検証環境
- インスタンス: db.t3.micro
- ストレージ: gp3 20GB（最大40GBまで自動拡張）
- 構成: Single-AZ
- 用途は軽負荷試験・動作確認のため、小規模構成とした。
- 同時接続数・クエリ数が少ない前提では十分な性能を持つ。

### 本番環境
- インスタンス: db.t3.small 以上を想定。
- ストレージ: gp3 20GB（最大50GBまで自動拡張）
- 構成: Multi-AZ（可用性向上）
- DBは単一ライター構成のため、システム全体のボトルネックになりやすい。
- そのため、検証環境よりも余裕を持ったスペックを選定する。


## 4. ECSとRDSの役割と、想定されるボトルネック
- ECS: アプリケーション処理（スケールアウト可能）
- RDS: データ処理（スケールアップ中心）
- 本構成では、以下の理由によりRDSがボトルネックになりやすい。

　　・一覧表示、検索で複数SQLが発行される
　　・ECSは横に増やせるが、RDSは単一インスタンス
　　・同時接続増加時にDB負荷が集中する


## 5. パフォーマンス悪化時の改善方針

### 1. CloudWatchでボトルネック特定
- ECS: CPUUtilization / MemoryUtilization / RunningTaskCount
- RDS: CPUUtilization / FreeStorageSpace
- ALB: HTTPCode_Target_5XX_Count
- Logs: アプリケーションエラー（ERROR / Exception）


### 2. ECSがボトルネックの場合
- タスク数の増加（AutoScaling調整）
- CPU/Memoryの増強（256 → 512 など）
- アプリケーションの処理見直し（不要処理削減）


### 3. RDSがボトルネックの場合
- インスタンスクラスのスケールアップ（micro → small → medium）
- インデックス追加
- N+1問題の解消
- SQLチューニング
- キャッシュ導入（アプリ・CloudFront）


### 4. 静的コンテンツ負荷対策
- CloudFrontキャッシュ最適化
- キャッシュTTL調整


## まとめ

- 本システムでは、CloudFrontにより静的コンテンツ配信を分離し、
ECSとRDSへの負荷を軽減している。

- ECSはAutoScalingによるスケールアウトで負荷増加に対応し、
RDSは適切なインスタンスサイズ選定とチューニングにより性能を確保する。

- また、CloudWatchによる継続的な監視を行い、
ボトルネックに応じたスケールアップ・スケールアウトを実施することで、
安定したパフォーマンスを維持する設計とした。
